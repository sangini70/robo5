# ENCODING_RULES.md

## 목적

이 프로젝트는 한글 UI 문자열을 포함하므로, 모든 코드 수정 작업에서 문자 인코딩 손상을 방지한다.

## 절대 원칙

- 모든 파일은 UTF-8 기준으로 유지한다.
- 한글 문자열을 PowerShell 자동 치환으로 대량 수정하지 않는다.
- 깨진 문자열을 그대로 저장하거나 commit하지 않는다.
- 의미를 모르는 한글 문자열을 임의로 추측해 바꾸지 않는다.
- UI 문구 수정 작업이 아닌 경우 한글 문자열은 건드리지 않는다.

## 금지

- 전체 파일 대상 일괄 replace 금지
- 인코딩 확인 없이 Set-Content 사용 금지
- 깨진 문자열을 다른 깨진 문자열로 임시 대체 금지
- 기존 한글 문구를 영어로 임의 변경 금지
- 로직 수정 중 UI 문구를 함께 정리하는 행위 금지

## 작업 전 확인

- 수정 대상 파일의 인코딩과 editor 저장 인코딩을 확인한다.
- 한글 문자열이 포함된 파일인지 확인한다.
- 한글 UI 문구를 수정해야 하는 작업인지 확인한다.

## 작업 중 규칙

- 한글 문자열은 최소 범위만 수정한다.
- 불가피하게 문자열을 수정할 경우 UTF-8로 저장한다.
- 기존 문구 의미가 불확실하면 임의 복구하지 말고 보고한다.
- 코드 로직 수정과 한글 문구 복구를 같은 작업에서 섞지 않는다.

## 작업 후 검증

- npm run build 실행
- git diff에서 한글 문자열 깨짐 여부 확인
- 화면에서 주요 한글 문구 확인
- 깨진 문자 패턴이 남아 있는지 검색
- 필요 시 my_gstack browser-operations / production-verification 기준으로 실제 rendering 확인

## 깨진 문자 의심 패턴

다음 문자가 UI 문자열에 나타나면 인코딩 손상 가능성이 높다.

- �
- æ
- ç
- å
- Ã
- Â
- 燦
- 吠
- 땐
- 벌씰
- ?가 연속된 문자열

## 완료 기준

- 한글 UI가 정상 표시된다.
- 기존 로직은 변경되지 않았다.
- 깨진 문자열이 git diff에 남아 있지 않다.
- git status clean 상태로 종료한다.
