# 0. 문서의 지위와 범위

## 목적

이 문서는 Robo5의 공식 Screen Architecture 기준서이다.

승인된 사용자 상태와 상태 전환을 어떤 화면 책임 체계가 담당하는지 정의한다.

이 문서는 페이지 목록, 사이트맵, UI 명세 또는 구현 명세가 아니다. `ROBO5_INFORMATION_ARCHITECTURE.md`가 정의한 정보공간과 `ROBO5_USER_FLOW.md`가 정의한 사용자 상태를 입력으로 받아 화면 책임을 배분한다.

## 이 문서가 소유하는 결정

- 승인된 사용자 상태를 담당할 화면 책임
- 하나의 화면 책임이 함께 수용할 수 있는 상태
- 독립 화면 책임을 검토해야 하는 조건
- 직접 진입 상태의 최초 담당 책임
- 정상·경계·빈·복구 상태의 담당 책임
- 화면 책임 사이의 인계 관계
- 인계 과정에서 보존할 사용자 문맥
- 각 화면 책임에서 사용자가 할 수 있어야 하는 판단
- Design과 Implementation으로 전달할 화면 책임 조건

## 이 문서가 소유하지 않는 결정

- Knowledge와 정보 객체의 원천 정의
- 정보 계층, Navigation, Discovery 및 관계의 정의
- 새로운 사용자 목적, 상태, 분기, 복귀 및 종료
- 색상, 서체, 크기, 여백 및 시각적 위계
- 화면 레이아웃과 구성 요소
- 버튼, 카드, 메뉴, 탭 및 상호작용 표현
- URL, slug, routing 및 파일 구조
- 코드, API, 데이터 모델 및 상태 관리
- 검색·추천·번역 알고리즘
- 콘텐츠 작성, 발행 및 운영 방식

# 1. Screen Architecture의 목적과 기본 원칙

## 목적

Screen Architecture는 승인된 User Flow가 디자인 또는 구현 단계에서 누락되거나 서로 다른 책임으로 혼합되지 않도록 화면 책임을 명확히 한다.

## 기본 단위

```text
승인된 사용자 상태
→ 담당 화면 책임
→ 상태 수용 방식
→ 필요한 사용자 판단
→ 다음 화면 책임으로의 인계
```

화면 책임은 화면의 형태가 아니다. 다음 책임의 묶음이다.

- 어떤 사용자 상태를 맡는가
- 어떤 사용자 목적을 지원하는가
- 어떤 IA 문맥을 받아야 하는가
- 사용자가 무엇을 확인하고 판단할 수 있어야 하는가
- 어떤 상태에서 책임을 종료하거나 다른 책임으로 넘기는가
- 인계할 때 어떤 문맥을 보존해야 하는가

## 기본 원칙

1. 화면 책임은 승인된 사용자 상태를 기준으로 정의한다.
2. 홈페이지를 거치지 않은 직접 진입도 정상 상태로 수용한다.
3. 하나의 사용자 상태와 하나의 독립 화면을 자동으로 동일시하지 않는다.
4. 하나의 정보공간과 하나의 독립 화면을 자동으로 동일시하지 않는다.
5. 현재 질문, 진입 문맥 및 학습 문맥을 불필요하게 잃지 않는다.
6. 성공, 종료, 경계, 빈 상태 및 복구 상태를 함께 수용한다.
7. 상위 Source of Truth에 실제 존재하는 상태와 관계만 사용한다.
8. 화면 수의 최소화보다 책임의 명확성을 우선한다.
9. 디자인 또는 구현 편의를 근거로 IA나 User Flow를 변경하지 않는다.
10. 실제로 존재하지 않는 로그인, 저장, 완료 기록, 진도, 재개 및 개인화 상태를 만들지 않는다.

## 권한 구분

| 문서 | 고유 책임 | Screen Architecture와의 관계 |
|---|---|---|

| `SOURCE_OF_TRUTH.md` | Source와 산출물의 단일성 및 계층 | 새로운 원천 정의와 중복 Source 생성을 금지하는 기준 |
| `COMMON_ROBO_CONTENT_ARCHITECTURE.md` | Knowledge 모델, 분류 및 관계 원칙 | 화면 책임이 참조하는 원천 Knowledge 구조 |
| `COMMON_ROBO_CONTENT_MAP.md` | Knowledge 구조의 시스템 실행 관계 | 승인된 화면 책임이 후속 실행 영역으로 전달되는 경계 |
| `ROBO5_INFORMATION_ARCHITECTURE.md` | 정보공간, 계층, Navigation, Discovery 및 복구 구조 | 화면 책임이 받아야 할 IA 문맥의 원천 |
| `ROBO5_USER_FLOW.md` | 사용자 목적, 상태 변화, 분기, 복귀, 성공 및 종료 | 화면 책임에 배정할 상태와 인계의 원천 |
| `ROBO5_SCREEN_ARCHITECTURE.md` | 사용자 상태와 화면 책임의 대응 체계 | 이 문서가 소유하는 공식 결정 |
| `ROBO5_MASTER_DESIGN_CONSTITUTION.md` | 브랜드, UX, 시각 표현 및 디자인 보호 원칙 | 승인된 화면 책임을 표현하는 후속 기준 |

## 권한 원칙

```text
Information Architecture
정보공간과 관계
        ↓
User Flow
사용자 목적과 상태 변화
        ↓
Screen Architecture
상태를 담당하는 화면 책임
        ↓
Design
화면 책임의 경험·시각 표현
        ↓
Implementation
승인된 화면 체계의 기술 실행
```

## 충돌 처리

1. 필요한 정보 객체나 관계가 IA에 없으면 이 문서에서 추가하지 않고 IA 변경 대상으로 돌려보낸다.
2. 필요한 사용자 목적이나 상태가 User Flow에 없으면 이 문서에서 추가하지 않고 User Flow 변경 대상으로 돌려보낸다.
3. 디자인 표현이 화면 책임을 누락하거나 혼합하면 화면 책임을 변경하지 않고 Design 단계에서 조정한다.
4. 구현 제약이 화면 책임과 충돌하면 구현 편의로 화면 책임을 변경하지 않는다.
5. 상위 문서와 해석이 충돌하면 상위 권한 문서의 정의를 우선한다.
6. 이 문서의 승인은 Design 또는 Implementation의 승인을 대신하지 않는다.

# 3. Screen Architecture 구성 단위와 표기 기준

## 화면 책임 기록 단위

| 항목 | 의미 |
|---|---|
| Screen Responsibility ID | 이 문서 안에서 화면 책임을 추적하는 식별자 |
| Screen Responsibility Name | 책임의 주된 목적을 나타내는 이름 |
| Supported User States | 책임이 수용하는 승인된 사용자 상태 |
| Supported User Goals | 책임이 지원하는 승인된 사용자 목적 |
| Entry States | 책임이 처음 또는 인계 후 받는 진입 상태 |
| Required IA Context | 책임 수행에 필요한 승인된 정보공간과 관계 |
| Required User Judgments | 사용자가 해당 책임 안에서 할 수 있어야 하는 판단 |
| State Variants | 같은 책임 안에서 구분해야 하는 정상·경계·빈·복구 상태 |
| Outgoing Responsibilities | 상태 전환 후 인계할 수 있는 화면 책임 |
| Context to Preserve | 인계 과정에서 유지해야 할 사용자 문맥 |
| Boundary Conditions | 책임을 종료하거나 다른 책임으로 넘기는 조건 |
| Recovery Responsibility | 정상 책임이 성립하지 않을 때 넘길 복구 책임 |
| Referenced User Flows | 책임의 근거가 되는 User Flow 식별자 |

Screen Responsibility ID는 코드, URL, route, 파일명 또는 구성 요소 이름을 의미하지 않는다.

## 상태 수용 방식

| 표기 | 의미 |
|---|---|
| 독립 책임 | 주된 사용자 목적과 정보 책임이 달라 별도 화면 책임으로 관리 |
| 동일 책임 변형 | 주된 목적과 정보 책임을 유지하며 같은 화면 책임 안에서 상태만 구분 |
| 보조 상태 | 원래 목적과 문맥을 유지한 채 짧게 확인한 후 복귀 가능한 상태 |
| 인계 상태 | 현재 책임을 종료하고 다른 화면 책임으로 넘기는 상태 |
| 복구 상태 | 정상 경로가 성립하지 않아 유효한 문맥으로 되돌리는 상태 |

# 4. 사용자 상태 분류 체계

이 장은 `ROBO5_USER_FLOW.md`의 상태를 화면 책임 배정 관점에서 정규화한다. 새로운 사용자 상태를 만들지 않는다.

| 상태군 | 포함하는 승인 상태 |
|---|---|
| 전체 범위 탐색 | 전체 정보공간 진입, 전문·보조 영역 이해, 관심 분야 발견 |
| 분야 탐색 | Category 범위 확인, Hub·Topic·Learning Path 발견 |
| 주제 학습 시작 | Hub 진입, Path 목적·범위·시작점 확인 |
| 개별 Knowledge 확인 | Article 또는 Node 직접 진입, 현재 질문과 핵심 Knowledge 확인 |
| 순차 학습 | Path 시작·중간·끝, Previous·Next, 선수·후속 Knowledge 확인 |
| 선택 확장 | Related Knowledge 확인, 상위 Hub 발견, Domain 간 승인 관계 이동 |
| 검색 재진입 | 구체적·넓은 검색, 결과 판단, 범위 축소, 결과 없음 |
| 용어 확인 | Glossary 독립 진입, Article 문맥의 짧은 정의, 상세 Knowledge 전환 |
| 언어 이동 | 언어 정보공간 진입, 대응 Knowledge 이동, 대응 Knowledge 없음 |
| 성공·종료 | 목적 달성, 종료, 복귀 또는 새로운 목적 전환 |
| 예외·복구 | 존재하지 않는 정보, 관계 부재, Path 단절, 예상과 다른 정보 도달 |

# 5. 화면 책임 분류 체계

## SR-01 전체 정보공간 진입 책임

- **담당 목적:** 구체적인 주제를 정하지 않은 사용자가 Robo5의 범위와 유효한 시작점을 판단하게 한다.
- **수용 상태:** 전체 정보공간 진입, 관심 분야 발견, 전문 영역과 보조 영역 구분.
- **필요 IA 문맥:** Domain, Category, 대표 Hub, 승인된 Learning Path.
- **필수 판단:** 현재 정보공간의 범위, 관심 분야, 다음 탐색 책임.
- **주요 인계:** SR-02, SR-03, SR-05, SR-07, SR-08.
- **근거 Flow:** UF-01, ML-01.

## SR-02 분야 탐색 책임

- **담당 목적:** 특정 분야 안에서 적절한 핵심 주제나 구체적 Knowledge를 찾게 한다.
- **수용 상태:** Category 직접 진입, 분야 범위 확인, Hub·Topic 구분, 구체적 질문으로 전환.
- **필요 IA 문맥:** Category, Hub, Topic, 대표 학습 시작점.
- **필수 판단:** 분야의 범위, 목적에 맞는 Hub 또는 Knowledge, 상위·인접 분야로의 복귀 가능성.
- **주요 인계:** SR-03, SR-04, SR-05, SR-07, SR-09.
- **근거 Flow:** UF-01, UF-02, SD-02.

## SR-03 주제 학습 책임

- **담당 목적:** 하나의 Hub와 Learning Path의 범위·시작점·현재 단계를 이해하게 한다.
- **수용 상태:** Hub 직접 진입, Path 시작, Article에서 상위 Hub 발견, Path 시작·중간·끝 확인.
- **필요 IA 문맥:** Knowledge Hub, Learning Path, 승인된 Node 순서, 선수·후속 관계.
- **필수 판단:** 주제 범위, 적절한 시작 위치, 현재 Path 문맥, 다음 학습 또는 종료.
- **주요 인계:** SR-04, SR-07, SR-09.
- **근거 Flow:** UF-03, UF-05, LP-01, LP-06, LP-07.

## SR-04 개별 Knowledge 확인 책임

- **담당 목적:** 하나의 질문과 핵심 Knowledge를 확인하고 현재 위치와 후속 선택을 판단하게 한다.
- **수용 상태:** Article·Node 직접 진입, 외부 진입, Path 중간 진입, 순차 이동 후 진입, Related 진입, Supporting Domain 진입.
- **필요 IA 문맥:** Knowledge Node, Article, Category·Hub·Topic 문맥, Previous·Next, Related, Glossary 관계.
- **필수 판단:** 현재 질문의 답, 목적 달성 여부, 상위 문맥, 선수·후속·관련 관계의 차이, 종료 가능성.
- **주요 인계:** SR-03, SR-04, SR-05, SR-06, SR-08, SR-09.
- **근거 Flow:** UF-04~UF-09, LP-02~LP-07, SD-01, SD-03, SD-05, GL-01, ML-02.

## SR-05 검색·재진입 책임

- **담당 목적:** 사용자의 표현을 유효한 Knowledge 문맥과 연결하고 탐색 또는 학습으로 재진입하게 한다.
- **수용 상태:** 내부 검색 진입, 구체적 질문 검색, 넓은 표현 검색, 검색 결과 판단, 재검색, 결과 없음.
- **필요 IA 문맥:** Search, Category, Hub, Topic, Knowledge, Glossary 및 승인된 표현 관계.
- **필수 판단:** 검색 의도와 결과의 관계, 빠른 확인과 깊은 학습의 차이, 범위 축소 또는 재검색 필요성.
- **주요 인계:** SR-02, SR-03, SR-04, SR-06, SR-09.
- **근거 Flow:** SD-01~SD-06, UF-01, UF-04, UF-09.

## SR-06 용어 확인 책임

- **담당 목적:** 용어의 짧은 의미를 확인하고 원래 문맥 복귀 또는 상세 학습 전환을 판단하게 한다.
- **수용 상태:** Glossary 독립 진입, Article 문맥의 용어 확인, 관련 용어 확인, 상세 Knowledge 없음.
- **필요 IA 문맥:** Glossary Term, 관련 분야, 상세 Knowledge 관계, 상위·하위·관련 용어 관계.
- **필수 판단:** 짧은 정의로 목적이 달성됐는지, 원래 문맥으로 복귀할지, 상세 Knowledge로 전환할지.
- **주요 인계:** SR-02, SR-03, SR-04, SR-05, SR-06, SR-09.
- **근거 Flow:** GL-01~GL-03, SD-04, UF-09.

## SR-07 목적별 학습 시작 책임

- **담당 목적:** 사용 목적 또는 수준에 맞는 승인된 학습 시작점과 Path를 찾게 한다.
- **수용 상태:** Learning Guide 진입, 전체·분야 정보공간에서 목적별 시작점 선택.
- **필요 IA 문맥:** Learning Guide와 승인된 Learning Path의 관계.
- **필수 판단:** 목적에 맞는 시작점, Guide와 실제 Path의 차이, 전환할 주제 학습 책임.
- **주요 인계:** SR-02, SR-03, SR-04, SR-09.
- **근거 Flow:** UF-01, UF-02, UF-03.

SR-07은 새로운 Knowledge 분류나 새로운 User Flow를 만들지 않는다. IA에 승인된 Learning Guide 책임을 기존 Flow의 시작점 판단에 적용한다.

## SR-08 언어 정보공간 진입 책임

- **담당 목적:** 일반 언어 정보공간 이동과 승인된 대응 Knowledge 이동을 구분하게 한다.
- **수용 상태:** 특정 언어 정보공간 진입, 일반 언어 이동, 대응 Knowledge 존재, 대응 Knowledge 없음.
- **필요 IA 문맥:** 언어별 Category·Hub·Search·Glossary·Learning Path, 승인된 대응 Knowledge 관계.
- **필수 판단:** 이동 대상의 언어 범위, 현재 Knowledge 문맥 유지 여부, 상위 정보공간 재진입 여부.
- **주요 인계:** SR-01~SR-07, SR-09.
- **근거 Flow:** ML-01~ML-03.

## SR-09 실패·빈 상태 복구 책임

- **담당 목적:** 사용할 수 없는 경로의 사실을 알리고 가장 가까운 유효 문맥으로 복구하게 한다.
- **수용 상태:** 검색 결과 없음, 존재하지 않는 정보, 상위 문맥 없음, 관계 부재, Path 단절, 번역본 없음, 변경된 Knowledge, 예상과 다른 정보 도달.
- **필요 IA 문맥:** 현재 유효한 상위·인접 정보공간과 승인된 복구 관계.
- **필수 판단:** 현재 사용할 수 없는 경로, 남아 있는 유효 문맥, 복귀·재검색·대체 탐색 중 가능한 선택.
- **주요 인계:** SR-01~SR-08 또는 정상 종료.
- **근거 Flow:** SD-06, GL-03, ML-03, User Flow 11장.

# 6. 사용자 상태와 화면 책임 대응 원칙

사용자 상태를 배정할 때 다음 순서로 판단한다.

1. 상태의 주된 사용자 목적을 확인한다.
2. 상태가 요구하는 IA 정보공간과 관계를 확인한다.
3. 직접 진입 가능한 상태인지 확인한다.
4. 원래 문맥 또는 진입 Path를 유지해야 하는지 확인한다.
5. 사용자가 해야 할 판단과 성공·종료 조건을 확인한다.
6. 기존 화면 책임 안에서 목적과 문맥을 유지할 수 있는지 판단한다.
7. 유지할 수 있으면 동일 책임 변형 또는 보조 상태로 수용한다.
8. 유지할 수 없으면 독립 화면 책임에 배정한다.
9. 정상 관계가 성립하지 않으면 SR-09에 복구 책임을 인계한다.

하나의 상태가 여러 책임과 관련되더라도 현재 상태의 주된 목적을 맡는 책임을 담당 책임으로 지정한다. 나머지는 인계 대상 또는 보조 책임으로 기록한다.

# 8. 탐색·학습·확인 화면 책임 체계

| 사용자 과업 | 담당 책임 | 책임의 시작 | 책임의 종료 또는 인계 |
|---|---|---|---|
| 넓은 분야 탐색 | SR-01, SR-02 | 전체 또는 분야 정보공간 진입 | 유효한 Hub·Knowledge·검색 시작점 선택 |
| 핵심 주제 발견 | SR-02 | Category 범위 확인 | Hub 또는 구체적 Knowledge 선택 |
| 주제 학습 시작 | SR-03 | Hub 또는 승인된 Path 진입 | 시작·현재 Node를 선택해 SR-04로 인계 |
| 개별 질문 확인 | SR-04 | Article·Node 진입 | 목적 달성 후 종료 또는 후속 책임 선택 |
| 상위 학습 문맥 발견 | SR-04 | 현재 Knowledge 확인 | SR-03에 Hub·Path 문맥을 인계 |
| 순차 학습 계속 | SR-04 | 현재 Path의 Node 확인 | 승인된 Previous·Next Node로 같은 책임을 이어감 |
| 관련 Knowledge 확장 | SR-04 | 현재 질문 확인 후 선택 확장 | 원래 문맥 복귀 또는 새 목적 책임으로 전환 |
| 목적별 시작점 발견 | SR-07 | 학습 목적 또는 수준 확인 | 승인된 SR-03 또는 SR-04로 인계 |
| 목적 달성 후 선택 | 현재 담당 책임 | Success State 도달 | 종료 또는 승인된 후속 책임으로 인계 |

탐색, 주제 학습, 개별 확인 및 복구는 서로 다른 주된 목적이다. 하나의 표현 체계에서 나타날 수 있더라도 화면 책임은 혼합하지 않는다.

# 9. Learning Path 화면 상태 체계

| Path 상태 | 담당 책임 | 수용 방식 | 필요한 판단 | 다음 책임 |

| Path 중간 | SR-04 | 동일 책임 변형 | 현재 단계, 선수·후속 관계 | SR-04, SR-03 또는 종료 |
| Path 끝 | SR-04 | 동일 책임 변형 | 승인된 Next 없음, 종료·Hub 복귀·Related 확장 | SR-03, SR-04 또는 종료 |
| 중간 Node 직접 진입 | SR-04 | 동일 책임 변형 | 현재 질문, Path와 단계, 선수 필요 여부 | SR-04, SR-03 또는 종료 |
| 선수 Knowledge 필요 | SR-04 | 인계 상태 | 승인된 선수 관계와 원래 단계 | 선수 Node의 SR-04 |

Previous·Next는 순차 학습 책임으로 유지하며 Related와 대체하거나 혼합하지 않는다.

실제 학습 완료 기록이 없으므로 완료율, 저장된 진도, 재개 및 미완료 상태를 만들지 않는다.

# 10. 검색 및 Discovery 화면 상태 체계

| 상태 | 담당 책임 | 수용 방식 | 필수 판단 | 인계 |
|---|---|---|---|---|
| 구체적인 질문 검색 | SR-05 | 동일 책임 변형 | 질문과 일치하는 Knowledge | SR-04 |
| 넓은 표현 검색 | SR-05 | 동일 책임 변형 | Category·Hub·Glossary 중 적합한 문맥 | SR-02, SR-03, SR-06 |
| 빠른 확인과 깊은 학습 분기 | SR-05 | 인계 상태 | 현재 목적의 깊이 | SR-04 또는 SR-03 |
| 검색 결과에서 상위 구조 재진입 | SR-05 | 인계 상태 | 적합한 상위 Category·Hub | SR-02 또는 SR-03 |
| 검색에서 Related Discovery 전환 | SR-05 | 인계 상태 | 현재 질문과 연결 이유 | SR-04 또는 SR-03 |
| 결과 없음 | SR-09 | 복구 상태 | 표현 조정, 상위 문맥 또는 재검색 | SR-02, SR-03, SR-05, SR-06 |

검색은 사용자의 명시적 표현에 대한 재진입 책임이고, Discovery는 현재 문맥에서 연결 이유가 있는 새로운 Knowledge 발견이다. 두 책임을 같은 상태로 취급하지 않는다.

검색 순위, 검색어 처리, 결과 계산 및 표현 방법은 이 문서에서 정의하지 않는다.

| 예상과 다른 정보 도달 | SR-09 | 현재 정보가 사용자의 예상과 다름 | 원래 문맥, 상위 문맥 또는 검색 | 적합한 SR-02~SR-06 |
| 필요한 화면 문맥 확인 불가 | SR-09 | 정상 책임에 필요한 문맥을 확정할 수 없음 | 확인 가능한 가장 가까운 유효 문맥 | 적합한 SR-01~SR-08 |

복구 책임은 실패를 숨기거나 존재하지 않는 정보와 관계를 만들지 않는다. 기술적 오류 원인과 처리 방법은 이 문서의 범위가 아니다.

# 14. 화면 책임 간 전환과 문맥 보존

## 보존 대상

화면 책임이 바뀔 때 해당 Flow에 존재하는 다음 문맥을 보존한다.

- 원래 사용자 목적
- 진입한 Category, Hub 또는 Path 문맥
- 현재 Knowledge와 현재 질문
- 현재 학습 단계
- 순차 이동인지 Related 확장인지에 대한 관계 의미
- 용어 확인 전 원래 학습 문맥
- 외부 직접 진입 여부
- 언어 이동 전후의 대응 Knowledge 관계
- 복구가 필요한 실패 상태와 사용할 수 없는 경로
- Success State 이후 시작된 새로운 목적

## 전환 원칙

1. 상태 전환과 분기 자체는 `ROBO5_USER_FLOW.md`를 따른다.
2. 이 문서는 새로운 이동 경로를 추가하지 않는다.
3. 같은 책임 안에서 상태가 바뀌어도 관계 의미가 달라지면 그 차이를 유지한다.
4. 순차 이동과 선택 확장을 같은 문맥으로 기록하지 않는다.
5. 보조 상태에서 원래 책임으로 복귀할 수 있으면 원래 목적을 유지한다.
6. 목적이 바뀌면 기존 Flow의 연장으로 위장하지 않고 새 책임으로 인계한다.
7. 원래 문맥이 유효하지 않으면 SR-09가 가장 가까운 승인 문맥으로 복구한다.

# 15. 동일 화면 상태와 독립 화면의 분리 기준

## 동일 화면 책임으로 수용할 수 있는 경우

- 주된 사용자 목적이 유지된다.
- 같은 IA 정보공간과 관계를 사용한다.
- 현재 질문과 진입 문맥을 유지할 수 있다.
- 성공·종료·복구 조건이 바뀌지 않는다.
- 별도의 정보 책임이 생기지 않는다.
- 기존 화면 책임 안에서 상태 차이를 설명할 수 있다.

## 독립 화면 책임을 검토해야 하는 경우

- 주된 사용자 목적이 변경된다.
- 독립적인 직접 진입을 정상 상태로 수용해야 한다.
- 다른 IA 정보공간을 담당한다.
- 성공·종료·복구 조건이 본질적으로 달라진다.
- 기존 책임에 포함하면 순차·관련·계층 관계의 의미가 혼합된다.
- 원래 문맥과 새 문맥의 책임을 동시에 유지할 수 없다.

## 판단 금지 기준

다음 항목만으로 독립 화면 여부를 결정하지 않는다.

- URL 또는 slug
- 파일이나 폴더 구조
- 구성 요소 재사용 여부
- 기존 route의 존재 여부
- 개발 난이도
- 화면 수를 줄이려는 목적
- 특정 시안의 레이아웃

# 17. Screen Architecture 검증 기준

| 검증 영역 | 검증 질문 |
|---|---|
| Coverage | 승인된 모든 User Flow 상태에 담당 화면 책임이 있는가? |
| Purpose | 각 화면 책임의 주된 사용자 목적이 하나의 일관된 책임으로 설명되는가? |
| Entry | 직접 진입과 내부 인계를 모두 정상 상태로 수용하는가? |
| Context | 책임 전환 뒤 원래 목적과 필요한 문맥이 유지되는가? |
| Boundary | IA, User Flow, Design 및 Implementation의 권한을 침범하지 않는가? |
| Separation | 탐색, 학습, 확인, 검색, 용어, 언어 및 복구 책임이 혼용되지 않는가? |
| State | 정상, 경계, 빈 상태 및 복구 상태가 구분되는가? |
| Continuity | 담당 책임이 없는 상태나 인계가 끊긴 상태가 없는가? |
| Recovery | 실패 상태가 가장 가까운 유효한 복구 책임으로 이어지는가? |
| Feasibility | 실제로 없는 완료 기록, 저장, 재개, 로그인 또는 개인화를 가정하지 않는가? |
| Traceability | 모든 화면 책임이 승인된 IA와 User Flow로 추적되는가? |
| Handoff | Design과 Implementation이 필요한 책임 조건을 구분해 받을 수 있는가? |

승인 조건은 위 검증 항목을 모두 충족하고, 화면 책임 매트릭스에 상충하는 중복 배정이 없는 상태다.

# 18. 변경 관리와 문서 연동

## 변경 순서

```text
화면 책임 문제 확인
→ 관련 사용자 상태 확인
→ User Flow 존재 여부 확인
→ IA 정보공간과 관계 확인
→ Screen Architecture 소유 결정인지 판정
→ 변경안 작성
→ 사용자 승인
→ 이 문서 반영
→ Design·Implementation 영향 전달
```

## 변경안 필수 항목

- 현재 문제
- 영향받는 Screen Responsibility ID
- 영향받는 User Flow와 사용자 상태
- 영향받는 IA 문맥
- 동일 책임 수용 또는 독립 책임 분리 판단의 변화
- 문맥 보존 영향
- 예외·복구 책임 영향
- Design과 Implementation에 전달할 영향

## 변경 규칙

1. 승인 전에는 화면 책임 변경을 확정하거나 구현하지 않는다.
2. 새로운 사용자 상태가 필요하면 `ROBO5_USER_FLOW.md` 변경 필요성을 먼저 검토한다.
3. 새로운 정보 객체나 관계가 필요하면 `ROBO5_INFORMATION_ARCHITECTURE.md`의 권한을 먼저 검토한다.
4. 디자인 표현 변경만 필요한 경우 이 문서를 변경하지 않는다.
5. 구현 방식 변경만 필요한 경우 이 문서를 변경하지 않는다.
6. 상위 Source of Truth 변경의 영향을 받으면 추적 매트릭스를 다시 검증한다.

# 19. Screen Architecture Anti-Patterns

다음 행위는 Screen Architecture 위반이다.

- IA의 정보공간 유형을 그대로 복사해 화면 목록으로 사용한다.
- 홈페이지, Category, Hub, Article을 한 줄의 고정 여정으로만 연결한다.
- 모든 사용자가 전체 정보공간 첫 진입점에서 시작한다고 가정한다.
- 하나의 사용자 상태마다 독립 화면을 강제한다.
- 하나의 화면 책임에 양립할 수 없는 목적을 합친다.
- 화면 책임을 URL, route, 파일명 또는 구성 요소 이름으로 정의한다.
- Category 탐색과 Hub 학습 책임을 혼용한다.
- Article 직접 진입을 예외 상태로 취급한다.
- Previous·Next와 Related를 같은 책임으로 혼합한다.
- Glossary의 짧은 확인과 상세 Knowledge 학습을 같은 목적 상태로 취급한다.
- 검색 결과와 Discovery를 같은 책임으로 취급한다.
- 정상 상태만 정의하고 경계·빈·복구 상태를 누락한다.
- 모든 오류를 SR-01로만 돌려보낸다.
- 존재하지 않는 로그인, 저장, 완료 기록, 진도, 재개 또는 개인화 상태를 만든다.
- 화면 책임을 Header, Footer, Sidebar, 카드, 버튼, 메뉴 또는 탭으로 설명한다.
- 색상, 서체, 크기, 여백, 레이아웃, 반응형 또는 시각적 강조를 규정한다.
- URL, routing, 코드, API, 데이터 구조 또는 상태 관리 방식을 규정한다.
- 검색·추천·번역 알고리즘을 정의한다.
- Design Constitution의 화면 구성을 다시 작성한다.
- 구현 편의를 이유로 User Flow를 변경한다.
- IA에 없는 정보 객체나 관계를 화면 단계에서 추가한다.

# 20. Design·Implementation Handoff

## Design에 전달하는 항목

- Screen Responsibility ID와 책임 목적
- 담당 사용자 상태와 사용자 목적
- 사용자가 판단할 수 있어야 하는 내용
- 정상·경계·빈·복구 상태
- 화면 책임 사이의 인계 관계
- 유지해야 할 사용자 문맥
- 순차 경로, 선택 확장 및 복구 경로의 의미 차이
- 숨기거나 혼합하면 안 되는 정보 책임

## Design에 전달하지 않는 항목

- 색상과 서체
- 크기와 여백
- 레이아웃과 그리드
- 구성 요소의 종류와 형태
- 상호작용 표현
- 반응형 배치
- 페이지별 시각 시안

이 항목은 `ROBO5_MASTER_DESIGN_CONSTITUTION.md`와 후속 디자인 작업이 담당한다.

## Implementation에 전달하는 항목

- Screen Responsibility ID
- 지원해야 할 승인 상태
- 책임 사이의 인계 관계
- 보존해야 할 사용자 문맥
- 예외·빈 상태·복구 책임
- IA와 User Flow 추적 관계

## Implementation에 전달하지 않는 항목

- URL과 routing
- 파일과 코드 구조
- 구성 요소 구조
- API와 데이터 모델
- 상태 관리 및 렌더링 방식
- 검색·추천·번역의 계산 방식

Implementation은 승인된 책임을 실행하되 화면 책임의 의미를 변경하지 않는다.

# 21. 최종 불변 원칙

```text
Information Architecture는
사용자가 이동할 수 있는 정보공간과 관계를 정의한다.

User Flow는
그 공간 안에서 사용자가 거치는 목적과 상태 변화를 정의한다.

Screen Architecture는
각 사용자 상태와 상태 전환을 담당하는
화면 책임 체계를 정의한다.

Design은
승인된 화면 책임을 사용자가 이해할 수 있게 표현한다.

Implementation은
승인된 화면 체계를 기술적으로 실행한다.
```

Screen Architecture는 페이지를 늘리는 문서가 아니다.

승인된 사용자 상태가 누락되지 않고, 서로 다른 목적과 관계가 혼합되지 않으며, 화면 책임 사이에서 필요한 문맥이 보존되도록 책임을 배분하는 문서다.

이 문서는 새로운 Knowledge, 정보공간, User Flow, 디자인 또는 구현 방식을 만들지 않는다.
