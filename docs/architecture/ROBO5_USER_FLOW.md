# 0. 문서의 지위와 범위

## 목적

이 문서는 Robo5의 공식 User Flow 기준서이다.

승인된 정보공간 안에서 사용자가 특정 목적을 달성하기 위해 거치는 진입, 이동, 선택, 분기, 복귀 및 종료 흐름을 정의한다.

이 문서는 다음 질문에 답한다.

- 사용자는 어떤 목적을 가지고 진입하는가?
- 사용자는 어떤 정보 상태에서 출발하는가?
- 목적 달성까지 어떤 논리적 단계를 거치는가?
- 어떤 조건에서 경로가 나뉘는가?
- 문맥을 벗어난 사용자는 어디로 복귀하는가?
- 목적 달성은 어떤 상태로 판단하는가?
- 목적 달성 뒤 어떤 후속 흐름으로 전환할 수 있는가?

## 적용 범위

이 문서의 Source of Truth 범위는 다음에 한정한다.

- 사용자 목적
- 진입 상태와 전제조건
- 목적별 상태 변화
- 정상·분기·대체·복구 흐름
- 성공·종료·후속 상태
- 흐름 간 충돌과 우선순위
- Screen Architecture로 전달할 흐름 요구

모든 흐름은 `ROBO5_INFORMATION_ARCHITECTURE.md`에 승인된 정보 객체와 관계 안에서만 성립한다.

## 소유하지 않는 결정

이 문서는 다음을 정의하지 않는다.

- Knowledge 객체의 원천 정의와 생성 기준
- 정보 계층, 정보공간 및 관계 유형
- Category, Hub, Topic, Node 또는 Article의 신규 구성
- Navigation 및 Discovery 체계의 신규 정의
- 개별 콘텐츠의 주제, 내용 및 작성 기준
- 화면 유형, 화면 수 및 화면별 기능 책임
- UI 구성 요소, 화면 배치, 시각적 위계 및 상호작용 표현
- 디자인 토큰, 반응형 규칙 및 브랜드 표현
- URL, slug, routing, 코드, API 및 데이터 구조
- 검색·추천·개인화 알고리즘
- CMS, 발행, 배포 및 분석 방식

User Flow에 필요한 정보 객체나 관계가 승인된 IA에 없다면 이 문서에서 임시로 만들지 않는다. 해당 권한 문서의 검토 대상으로 돌려보낸다.

# 1. User Flow의 목적과 기본 원칙

## 목적

Robo5의 사용자 이동을 페이지 순서가 아니라 목적 달성을 위한 상태 변화로 관리한다.

## 기본 모델

```text
사용자 목적
→ 진입
→ 현재 문맥 확인
→ 목적에 맞는 Knowledge 접근
→ 선택 또는 분기
→ 목적 달성 판단
→ 종료·복귀·확장
```

이 모델은 화면 순서나 구현 순서를 의미하지 않는다.

## 원칙

### 목적 중심

모든 Flow는 하나의 명확한 사용자 목적에서 시작한다. 페이지 방문 자체를 목적으로 간주하지 않는다.

### 진입점 독립성

홈페이지를 거치지 않은 진입을 예외로 취급하지 않는다. 외부 검색, AI 인용, 공유 링크 및 개별 Knowledge 직접 진입도 정상 진입이다.

### 현재 문맥 유지

사용자는 이동 중 자신이 무엇을 확인하고 있으며 어떤 학습 문맥에 있는지 잃지 않아야 한다.

### 선택 가능성

빠른 확인, 순차 학습, 관련 Knowledge 확장 및 종료는 서로 다른 목적이다. 하나의 경로를 모든 사용자에게 강제하지 않는다.

### 관계의 정합성

모든 이동은 승인된 계층, 순차, 문맥, 횡단 또는 대응 관계를 따른다. 관계의 의미를 바꾸어 사용하지 않는다.

### 복귀 가능성

보조 정보나 다른 Knowledge를 확인하기 위해 원래 흐름을 벗어난 사용자는 유효한 원래 문맥 또는 가장 가까운 상위 문맥으로 복귀할 수 있어야 한다.

### 실패 후 복구

정상 경로가 성립하지 않을 때 존재하지 않는 관계를 암시하지 않는다. 사용할 수 있는 가장 가까운 유효 경로로 전환한다.

### 명확한 시작과 종료

모든 Flow는 진입 상태, 성공 상태 및 종료 상태를 구분한다. 계속 탐색하는 것만을 성공으로 간주하지 않는다.

### 실제 상태만 사용

현재 시스템에 존재하지 않는 학습 완료 기록, 저장, 개인화 또는 재개 기능을 전제하지 않는다.

## 규칙

1. 사용자 목적은 한 문장으로 표현할 수 있어야 한다.
2. 하나의 Flow에 서로 다른 목적이 포함되면 결정 지점에서 별도 Flow로 전환한다.
3. 사용자가 현재 질문에 대한 답을 확인한 뒤 종료할 수 있어야 한다.
4. 후속 이동은 선택 가능한 경로이며 강제 연속 소비가 아니다.
5. 흐름은 사용자가 무엇을 알고 판단할 수 있게 되는지를 기술한다.
6. 특정 버튼, 카드, 메뉴, 화면 또는 기술 동작으로 흐름을 설명하지 않는다.

# 2. Source of Truth 권한 경계

## 목적

User Flow가 기존 Source of Truth를 반복하거나 다른 문서의 책임을 선점하지 않게 한다.

## 권한 구분

| 문서 | 고유 책임 | User Flow와의 관계 |
|---|---|---|
| `GEO_TECHNICAL_CONSTITUTION.md` | Knowledge·Learning Path에 관한 상위 철학 | 흐름 판단의 상위 원칙 |
| `SOURCE_OF_TRUTH.md` | Source 계층과 단일성 | 흐름이 새로운 데이터 Source를 만들지 않게 하는 기준 |
| `COMMON_ROBO_CONTENT_ARCHITECTURE.md` | Knowledge 객체·분류·관계 원칙 | 흐름이 사용하는 원천 Knowledge 모델 |
| `COMMON_ROBO_CONTENT_MAP.md` | Knowledge 구조의 시스템 실행 관계 | 승인된 흐름 이후의 실행 영역 |
| `ROBO5_INFORMATION_ARCHITECTURE.md` | 정보공간·계층·Navigation·Discovery·탐색 복구 | 사용자가 이동할 수 있는 승인 공간 |
| `ROBO5_USER_FLOW.md` | 목적별 진입·상태 변화·분기·복귀·종료 | 승인 공간 안의 사용자 이동 기준 |
| `ROBO5_SCREEN_ARCHITECTURE.md` | 상태와 이동을 담당하는 화면 체계 | 승인된 Flow를 화면 책임으로 변환하는 후속 문서 |
| `ROBO5_MASTER_DESIGN_CONSTITUTION.md` | 브랜드·UX·시각 표현 원칙 | 승인된 구조와 흐름의 표현 기준 |
| 구현 문서 | 기술 실행 | 승인된 구조·흐름·화면 책임의 실행 영역 |

## 권한 원칙

```text
Information Architecture는
사용자가 이동할 수 있는 Knowledge 공간을 정의한다.

User Flow는
그 공간 안에서 사용자가 목적을 달성하기까지의
상태 변화와 이동 조건을 정의한다.

Screen Architecture는
각 상태와 이동을 담당하는 화면 체계를 정의한다.
```

## 충돌 처리

1. 정보 객체의 의미 또는 관계가 불명확하면 Content Architecture와 IA를 먼저 확인한다.
2. 이동 가능한 경로가 IA에 없으면 User Flow에서 새로 만들지 않는다.
3. 화면 필요성이 발견되면 화면을 확정하지 않고 Screen Architecture 전달 항목으로 기록한다.
4. 기술 제약은 User Flow 원칙으로 만들지 않고 구현 문서의 검토 대상으로 전달한다.
5. 상위 문서와 충돌하는 흐름은 승인된 User Flow로 간주하지 않는다.
6. User Flow의 승인은 IA, Screen Architecture, Design 또는 Implementation 변경 승인을 대신하지 않는다.

# 3. User Flow 구성 단위와 표기 기준

## 목적

모든 Flow를 같은 논리 구조로 기록하고 검증할 수 있게 한다.

## 필수 구성 단위

| 항목 | 책임 |
|---|---|
| Flow ID | 흐름을 식별하는 고유 표기 |
| Flow Name | 사용자 목적을 드러내는 흐름 이름 |
| User Goal | 사용자가 달성하려는 단일 목적 |
| Entry State | Flow가 시작되는 정보 상태 |
| Preconditions | Flow 성립에 필요한 승인 관계 |
| Main Flow | 목적 달성까지의 대표 상태 변화 |
| Decision Points | 사용자 의도 또는 정보 존재 여부에 따른 분기 |
| Alternative Flow | 주경로 외에 목적을 달성할 수 있는 유효 경로 |
| Recovery Flow | 실패 또는 단절 후 유효한 문맥으로 복귀하는 경로 |
| Success State | 사용자 목적이 충족된 논리적 상태 |
| Exit State | 현재 Flow가 끝나는 상태 |
| Next Possible Flow | 목적 변경 또는 학습 확장으로 전환 가능한 후속 Flow |
| Referenced IA Objects | 흐름이 참조하는 승인된 정보 객체와 관계 |
| Screen Architecture Handoff | 후속 문서가 지원해야 할 사용자 상태와 판단 |

## 상태 표기

Flow는 다음 상태를 구분한다.

- 시작
- 현재 문맥 확인
- Knowledge 확인
- 사용자 선택
- 조건 분기
- 정상 이동
- 대체 이동
- 원래 문맥 복귀
- 성공
- 종료
- 후속 Flow 전환

## 작성 규칙

1. Flow ID는 문서 내 식별을 위한 표기이며 코드나 데이터 식별자가 아니다.
2. 상태는 화면명이나 UI 요소가 아니라 사용자의 정보 상태로 작성한다.
3. 분기 조건은 사용자 목적 또는 승인된 정보의 존재 여부로 작성한다.
4. Main Flow와 Alternative Flow는 동일한 관계를 서로 다른 의미로 사용하지 않는다.
5. Recovery Flow는 실패 이유와 복귀 가능한 유효 문맥을 함께 명시한다.
6. Success State와 Exit State를 동일시하지 않는다. 성공 후 확장할 수도 있고 성공 없이 복구 Flow로 전환할 수도 있다.
7. 이 문서의 기준 Flow는 1장과 5장의 공통 원칙 및 해당 장의 공통 전제조건을 상속한다.
8. 기준 Flow에서 공통 항목을 반복하지 않을 수 있으나, 개별 Flow를 별도 승인 대상으로 작성할 때는 필수 구성 단위를 모두 확인한다.

# 4. 사용자 목적과 과업 체계

## 목적

사용자 흐름을 화면이나 페이지 유형이 아니라 사용자가 현재 수행하려는 과업으로 분류한다.

## 핵심 과업군

| 과업군 | 사용자 목적 |
|---|---|
| 빠른 확인 | 하나의 개념이나 질문에 필요한 정보를 확인한다 |
| 분야 탐색 | 관심 분야 안에서 적절한 주제와 시작점을 찾는다 |
| 주제 학습 시작 | 하나의 Hub와 시작 Knowledge를 찾아 학습을 시작한다 |
| 순차 학습 | 현재 단계의 선수·후속 관계를 이해하고 다음 단계로 이동한다 |
| 목적형 정보 탐색 | 비교·원인·방법·판단 등 현재 질문에 맞는 Knowledge를 찾는다 |
| 관련 Knowledge 확장 | 현재 Knowledge와 관계 있는 다른 개념으로 범위를 넓힌다 |
| 용어 확인 | 모르는 용어의 의미를 확인하고 원래 문맥을 유지한다 |
| 검색 재진입 | 사용자의 표현을 유효한 Knowledge 구조와 연결한다 |
| 실패 복구 | 막힌 경로에서 유효한 상위 또는 인접 문맥을 찾는다 |
| 언어 이동 | 의도한 언어 정보공간 또는 대응 Knowledge로 이동한다 |

## 규칙

1. 사용자 유형은 인구통계학적 Persona가 아니라 현재 과업으로 구분한다.
2. 동일한 사용자가 한 세션에서 여러 과업으로 전환할 수 있다.
3. 과업이 변경되면 기존 Flow를 억지로 연장하지 않고 적절한 후속 Flow로 전환한다.
4. 빠른 확인을 깊은 학습보다 낮은 가치의 행동으로 취급하지 않는다.
5. 탐색의 지속 시간이나 방문 수를 성공 조건으로 사용하지 않는다.

# 5. 공통 진입·이동·종료 모델

## 목적

모든 세부 Flow가 공유하는 기본 상태 전이를 정의한다.

## 공통 흐름

```text
진입
→ 현재 정보 확인
→ 위치와 문맥 이해
→ 목적에 맞는 경로 선택
→ Knowledge 확인 또는 학습
→ 목적 달성 판단
→ 종료·복귀·확장 중 선택
```

## 진입 상태

다음 진입은 모두 정상 상태다.

- 전체 정보공간에서 시작
- 분야 정보공간에서 시작
- 주제 학습공간에서 시작
- 개별 Knowledge에서 직접 시작
- 내부 검색 결과에서 시작
- 금융용어에서 시작
- Related 관계를 통해 시작
- Previous·Next 관계를 통해 시작
- 외부 검색, AI 인용 또는 공유 링크에서 시작
- 오류 또는 빈 상태에서 시작
- 특정 언어 정보공간에서 시작

## 이동 원칙

1. 상위 공간 진입 사용자는 범위를 좁혀 적절한 Knowledge에 도달한다.
2. 하위 Knowledge 직접 진입 사용자는 현재 질문을 먼저 확인한 뒤 상위 문맥이나 후속 경로를 선택한다.
3. 빠른 확인 목적은 답을 확인한 상태에서 성공할 수 있다.
4. 깊은 학습 목적은 현재 위치와 학습 방향을 이해한 상태에서 다음 단계로 이어질 수 있다.
5. 보조 경로로 이동할 때 원래 목적과 현재 문맥을 구분한다.
6. 목적이 바뀌면 새로운 Flow로 전환한다.

## 종료 상태

현재 Flow는 다음 중 하나로 끝난다.

- 목적 달성 후 종료
- 원래 학습 문맥으로 복귀
- 상위 정보공간으로 이동
- 다음 학습 단계로 전환
- Related Knowledge 탐색으로 전환
- 새로운 검색으로 전환
- 다른 언어 정보공간으로 전환
- 복구 가능한 문맥으로 전환

# 6. 핵심 사용자 흐름

## 목적

Robo5에서 반복적으로 발생하는 대표 사용자 이동을 목적별로 정의한다.

## UF-01 관심 분야 발견

**User Goal**  
전체 Knowledge 공간에서 관심 분야의 유효한 시작점을 찾는다.

**Entry State**  
사용자는 Robo5의 전체 범위 안에 있지만 구체적인 주제는 정하지 않았다.

**Preconditions**  
대표 Domain, Category, Hub 또는 Learning Path가 IA에 승인되어 있어야 한다.

**Main Flow**

```text
전체 정보공간 진입
→ 전문 영역과 보조 영역의 차이 이해
→ 관심 분야 선택
→ 분야 범위 확인
→ 적절한 Hub·Topic·Learning Path 발견
→ 주제 학습 또는 세부 탐색으로 전환
```

**Decision Points**

- 구체적인 질문이 생기면 검색 또는 직접 Knowledge 탐색으로 전환한다.
- 학습 목적이 분명해지면 UF-03으로 전환한다.

**Success State**  
사용자가 자신의 관심에 맞는 유효한 분야 또는 학습 시작점을 찾았다.

**Exit State**  
분야 탐색, 주제 학습, 검색 또는 종료 중 하나로 전환한다.

## UF-02 분야에서 Hub 발견

**User Goal**  
관심 분야 안에서 학습할 핵심 주제를 찾는다.

**Entry State**  
사용자는 특정 Category의 범위를 확인하고 있다.

**Preconditions**  
Category와 Hub·Topic의 관계가 IA에 승인되어 있어야 한다.

**Main Flow**

```text
분야 범위 확인
→ 대표 주제와 세부 탐색 문맥 구분
→ 목적에 맞는 Hub 선택
→ Hub의 범위와 학습 방향 확인
→ 학습 시작점 선택
```

**Alternative Flow**  
구체적인 질문이 이미 있으면 해당 Knowledge로 직접 이동한다.

**Success State**  
사용자가 학습 목적에 맞는 Hub 또는 구체적인 Knowledge를 찾았다.

## UF-03 Hub에서 학습 시작

**User Goal**  
하나의 주제를 적절한 시작점부터 학습한다.

**Entry State**  
사용자는 특정 Hub에 진입했다.

**Preconditions**  
Hub의 범위와 승인된 Learning Path가 존재해야 한다.

**Main Flow**

```text
주제 범위 확인
→ Learning Path의 목적과 범위 이해
→ 시작 Knowledge 확인
→ 시작 단계 학습
→ 다음 단계 이동 또는 종료
```

**Decision Points**

- 선수 Knowledge가 필요하면 해당 Knowledge를 먼저 확인한다.
- 현재 질문만 확인하려면 빠른 확인 Flow로 전환한다.

**Success State**  
사용자가 적절한 시작 Knowledge를 찾고 현재 학습 방향을 이해했다.

## UF-04 외부에서 Article 직접 진입

**User Goal**  
외부에서 진입한 질문의 답을 확인하고 필요한 후속 경로를 선택한다.

**Entry State**  
사용자는 외부 검색, AI 인용, 공유 링크 또는 다른 외부 경로를 통해 개별 Article에 진입했다.

**Preconditions**  
Article이 승인된 Node와 상위 문맥에 연결되어 있어야 한다.

**Main Flow**

```text
개별 Article 진입
→ 현재 질문과 핵심 Knowledge 확인
→ 현재 위치와 상위 문맥 이해
→ 목적 달성 여부 판단
├─ 달성 → 종료 또는 선택 확장
└─ 미달성 → 선수·후속·관련 Knowledge 중 목적에 맞는 경로 선택
```

**Recovery Flow**  
상위 문맥이나 유효한 관계가 없으면 가장 가까운 Category, 검색 또는 유효한 Knowledge 공간으로 전환한다.

**Success State**  
사용자가 현재 질문의 답을 확인하고 자신의 다음 선택을 이해했다.

## UF-05 Article에서 상위 Hub 발견

**User Goal**  
개별 질문을 더 넓은 주제 학습으로 확장한다.

**Entry State**  
사용자는 Article의 핵심 Knowledge를 확인한 상태다.

**Preconditions**  
Article과 상위 Hub의 관계가 승인되어 있어야 한다.

**Main Flow**

```text
현재 Knowledge 확인
→ 상위 주제 문맥 확인
→ Hub의 범위와 Learning Path 확인
→ 시작점·현재 단계·선수 단계 중 목적에 맞는 위치 선택
→ 주제 학습으로 전환
```

**Success State**  
사용자가 현재 Article의 상위 학습 문맥과 적절한 진입 위치를 찾았다.

## UF-06 Previous·Next 순차 이동

**User Goal**  
현재 Learning Path의 순서에 따라 학습을 이어간다.

**Entry State**  
사용자는 특정 Path의 현재 Node를 학습하고 있다.

**Preconditions**  
현재 Path 문맥과 Previous·Next 관계가 승인되어 있어야 한다.

**Main Flow**

```text
현재 단계 확인
→ 선수·후속 단계 구분
→ 현재 목적에 맞는 방향 선택
→ 선택한 단계로 이동
→ 새로운 현재 단계와 Path 문맥 확인
```

**Alternative Flow**  
순차 학습보다 개념 확장이 필요하면 UF-07로 전환한다.

**Success State**  
사용자가 순차 관계의 의미를 유지한 채 이전 또는 다음 단계로 이동했다.

## UF-07 Related Knowledge 확장

**User Goal**  
현재 Knowledge와 문맥적으로 관련된 다른 Knowledge로 이해 범위를 넓힌다.

**Entry State**  
사용자는 현재 질문을 확인했으며 선택적인 확장을 원한다.

**Preconditions**  
Related 관계와 연결 이유가 IA에 승인되어 있어야 한다.

**Main Flow**

```text
현재 Knowledge 확인
→ 관련 관계의 의미 확인
→ 확장 목적에 맞는 Knowledge 선택
→ Related Knowledge 확인
→ 원래 문맥 복귀 또는 새 학습 목적 전환
```

**Decision Points**

- 원래 질문을 보완하는 목적이면 확인 후 복귀한다.
- 새로운 주제를 학습하려는 목적이면 별도 Hub 또는 Learning Path Flow로 전환한다.

**Success State**  
사용자가 순차 관계와 혼동하지 않고 관련 Knowledge로 이해를 확장했다.

## UF-08 Supporting Domain에서 Primary Domain으로 이동

**User Goal**  
보조 Knowledge를 핵심 전문 영역의 학습 문맥과 연결한다.

**Entry State**  
사용자는 Supporting Domain의 Knowledge를 확인하고 있다.

**Preconditions**  
Primary Domain과 연결되는 승인된 문맥 또는 횡단 관계가 있어야 한다.

**Main Flow**

```text
보조 Knowledge 확인
→ 핵심 전문 영역과의 연결 이유 확인
→ 관련 Primary Knowledge 선택
→ Primary Domain의 현재 문맥 확인
→ 주제 학습 또는 질문 확인으로 전환
```

**Recovery Flow**  
승인된 연결 관계가 없으면 임의의 Primary Knowledge를 제시하지 않고 현재 Domain의 상위 문맥 또는 검색으로 전환한다.

**Success State**  
사용자가 두 Domain의 역할을 혼동하지 않고 승인된 관계를 따라 이동했다.

## UF-09 목적 달성 후 종료 또는 후속 탐색

**User Goal**  
현재 질문이 해결된 뒤 종료할지 학습을 확장할지 선택한다.

**Entry State**  
현재 Flow의 Success State에 도달했다.

**Main Flow**

```text
목적 달성 확인
→ 추가 목적 존재 여부 판단
├─ 없음 → 현재 Flow 종료
├─ 순차 학습 → Next Flow 전환
├─ 주제 확장 → Related 또는 Hub Flow 전환
├─ 용어 확인 → Glossary Flow 전환
└─ 새 질문 → 검색 Flow 전환
```

**Success State**  
사용자가 현재 목적의 종료와 새로운 목적의 시작을 구분했다.

# 7. Learning Path 흐름

## 목적

승인된 학습 방향이 실제 사용자 이동에서 어떻게 이어지고 분기되는지 정의한다.

## 기본 방향

현재 프로젝트의 승인된 기본 Flow는 다음과 같다.

```text
정의 → 원인 → 방법 → 비교 → 판단 → 실행
```

이 문서는 개별 Node의 Flow 값을 새로 지정하지 않는다. 승인된 Path를 사용자 상태 변화에 적용한다.

## LP-01 Path 시작

```text
Hub 진입
→ Path 목적과 전체 범위 확인
→ 시작 Node 확인
→ 시작 Node 학습
→ 다음 단계 또는 종료 선택
```

성공 상태는 Path의 목적과 시작점을 이해한 상태다.

## LP-02 중간 Node 직접 진입

```text
중간 Node 진입
→ 현재 질문 확인
→ 현재 Path와 단계 확인
→ 선수 Knowledge 필요 여부 판단
├─ 필요 → 선수 Knowledge 확인
└─ 불필요 → 현재 단계 유지
→ 후속 단계 또는 종료 선택
```

중간 진입 사용자를 반드시 시작 Node로 돌려보내지 않는다.

## LP-03 선수 Knowledge 확인

```text
현재 Knowledge 이해 부족 확인
→ 승인된 선수 Knowledge 확인
→ 선수 Knowledge 학습
→ 원래 단계 복귀 또는 Path를 선수 단계부터 계속
```

복귀와 Path 재시작은 사용자의 현재 목적에 따라 구분한다.

## LP-04 다음 Knowledge 이동

```text
현재 단계 목적 달성
→ 승인된 다음 단계 확인
→ 다음 단계 학습 의사 판단
├─ 계속 → 다음 Node로 이동
└─ 중단 → 현재 Flow 종료
```

실제 완료 기록이 없으므로 다음 Node로 이동하지 않았다는 이유만으로 미완료 상태를 저장하거나 추정하지 않는다.

## LP-05 필수 순서와 선택 확장

```text
현재 단계 확인
→ 이동 관계의 의미 구분
├─ 선수·후속 관계 → 순차 학습
└─ Related 관계 → 선택 확장
```

Previous·Next와 Related를 같은 이동으로 취급하지 않는다.

## LP-06 Path 시작·중간·끝

- 시작 상태: 선행 Node가 없는 승인된 시작점
- 중간 상태: 선수와 후속 관계 중 하나 이상을 가진 현재 단계
- 끝 상태: 승인된 후속 Node가 없는 현재 Path의 마지막 단계

Path 끝은 전체 Knowledge 탐색의 끝을 의미하지 않는다. 사용자는 종료, Hub 복귀 또는 Related 확장 중 하나를 선택할 수 있다.

## LP-07 여러 Path에 속한 Node

```text
Node 진입
→ 현재 진입 Path 확인 가능 여부 판단
├─ 확인 가능 → 해당 Path 문맥 우선
└─ 확인 불가 → 원천 소속과 가능한 Path를 왜곡 없이 확인
→ 목적에 맞는 Path 선택
```

하나의 Path를 임의의 기본값으로 확정하지 않는다.

# 8. 검색 및 Discovery 흐름

## 목적

검색과 발견을 단순 결과 소비가 아니라 유효한 Knowledge 구조로 재진입하는 과정으로 정의한다.

## SD-01 구체적 질문 검색

```text
구체적인 질문 형성
→ 검색 결과 확인
→ 질문과 일치하는 Knowledge 선택
→ 핵심 답 확인
→ 종료·상위 Hub·후속 Knowledge 중 선택
```

성공 상태는 사용자의 표현이 적절한 Knowledge와 연결된 상태다.

## SD-02 넓은 표현 검색

```text
넓은 분야 또는 주제 검색
→ 결과 범위 확인
→ Category·Hub·Glossary 중 목적에 맞는 문맥 선택
→ 범위 축소
→ 구체적인 Knowledge 또는 학습 시작점 도달
```

넓은 검색을 관련성 없는 다수의 Article 탐색으로 끝내지 않는다.

## SD-03 빠른 확인과 깊은 학습 분기

```text
검색 결과 진입
→ 핵심 정보 확인
→ 사용자 목적 판단
├─ 빠른 확인 → 성공 후 종료
└─ 깊은 학습 → 상위 Hub 또는 Learning Path로 전환
```

## SD-04 Glossary 결과에서 상세 Knowledge 이동

```text
용어 결과 확인
→ 짧은 정의로 목적 달성 여부 판단
├─ 달성 → 종료 또는 원래 문맥 복귀
└─ 미달성 → 승인된 상세 Knowledge 확인
```

## SD-05 검색에서 Discovery로 전환

```text
검색 결과 확인
→ 현재 질문과 인접한 학습 필요 발견
→ 연결 이유가 있는 Knowledge 선택
→ Related 또는 Hub Flow로 전환
```

검색 결과와 Discovery 대상의 역할을 혼합하지 않는다.

## SD-06 검색 결과 없음

```text
결과 없음 인식
→ 표현 범위 또는 용어 확인
→ 상위 Category·Hub·Glossary 확인
→ 재검색 또는 대체 탐색
```

존재하지 않는 결과나 관계를 암시하지 않는다.

## 규칙

1. 검색 순위, 검색어 처리 및 추천 계산 방식은 이 문서에서 정의하지 않는다.
2. 같은 Knowledge의 중복 표현이 서로 다른 독립 결과로 오인되지 않게 한다.
3. 검색 실패는 종료만을 의미하지 않으며 유효한 구조로의 재진입 기회를 제공한다.
4. Discovery는 현재 질문을 방해하지 않고 선택적 확장으로 작동한다.

# 9. 금융용어 확인과 문맥 복귀 흐름

## 목적

모르는 용어를 확인하는 과정이 원래 학습 목적과 현재 문맥을 끊지 않게 한다.

## GL-01 Article 문맥에서 용어 확인

```text
Article 학습
→ 모르는 용어 발견
→ 짧은 정의 확인
→ 추가 설명 필요 여부 판단
├─ 불필요 → 원래 Article 문맥 복귀
└─ 필요 → 상세 Knowledge 확인
           → 원래 문맥 복귀 또는 새 학습 Flow 전환
```

## GL-02 독립적인 Glossary 진입

```text
Glossary 진입
→ 용어 확인
→ 짧은 정의로 목적 달성 여부 판단
├─ 달성 → 종료
├─ 상세 이해 필요 → 상세 Knowledge 이동
└─ 관련 용어 탐색 필요 → 승인된 관련 용어 이동
```

## GL-03 상세 Knowledge 없음

```text
용어 정의 확인
→ 상세 Knowledge 필요
→ 승인된 상세 관계 없음 확인
→ 현재 정의에서 종료
   또는 상위 용어 문맥·검색으로 전환
```

상세 Knowledge가 없는 상태에서 존재하는 것처럼 암시하지 않는다.

## 문맥 전환 기준

- 용어 확인만 끝나면 원래 Flow로 복귀한다.
- 용어 자체를 깊게 학습하려는 목적이 생기면 새로운 학습 Flow로 전환한다.
- 관련 용어로 이동하며 질문이 바뀌면 기존 Flow의 연장이 아니라 새 목적의 시작으로 기록한다.

# 10. 다국어 정보공간 이동 흐름

## 목적

일반적인 언어 정보공간 이동과 동일 Knowledge의 대응 콘텐츠 이동을 구분한다.

## ML-01 언어 정보공간 진입

```text
현재 언어 정보공간 확인
→ 의도한 언어 확인
→ 해당 언어 정보공간으로 이동
→ 이동한 공간의 범위와 문맥 확인
```

## ML-02 대응 Knowledge가 있는 이동

```text
현재 Knowledge 확인
→ 다른 언어의 대응 Knowledge 존재 확인
→ 대응 Knowledge로 이동
→ 동일 Knowledge 문맥 유지 여부 확인
```

대응 관계가 승인된 경우에만 같은 Knowledge의 언어 이동으로 취급한다.

## ML-03 대응 Knowledge가 없는 이동

```text
다른 언어 이동 의사 확인
→ 대응 Knowledge 없음 확인
→ 해당 언어의 상위 정보공간 또는 검색으로 이동
→ 새 문맥에서 탐색 재개
```

유사한 콘텐츠를 번역본으로 대체하지 않는다.

## 규칙

1. 언어 이동 뒤 기존 학습 위치가 유지되는지 여부를 명확히 구분한다.
2. 대응 콘텐츠가 없으면 동일 Path의 같은 단계에 도달했다고 간주하지 않는다.
3. 언어 선택 방식, 주소 체계 및 구현 방식은 이 문서에서 정의하지 않는다.

---

# 11. 예외·실패·복구 흐름

## 목적

정상 경로가 성립하지 않을 때 승인되지 않은 관계를 만들지 않고 가장 가까운 유효 문맥으로 복귀하게 한다.

## 공통 복구 모델

```text
현재 상태 인식
→ 사용할 수 없는 경로 확인
→ 가장 가까운 유효 문맥 확인
→ 복귀·재검색·대체 탐색 중 선택
→ 새 Flow 진입
```

## 복구 기준

| 예외 상태 | 복구 원칙 |
|---|---|
| 검색 결과 없음 | 표현 범위를 조정하거나 Category·Hub·Glossary에서 재진입 |
| 존재하지 않는 정보 | 의미상 가장 가까운 유효 상위 또는 인접 문맥으로 이동 |
| 상위 문맥 없음 | 임의의 상위 관계를 만들지 않고 검색 또는 유효한 분야 문맥으로 이동 |
| Previous 없음 | 현재 Path의 시작 상태로 이해하고 현재·다음 단계 중 선택 |
| Next 없음 | 현재 Path의 끝 상태로 이해하고 종료·Hub 복귀·Related 확장 중 선택 |
| Related 없음 | 순차 관계나 인기 Knowledge로 대체하지 않고 현재 Flow를 종료할 수 있게 함 |
| Learning Path 단절 | 존재하는 단계만 인정하고 유효한 Hub 또는 검색으로 복귀 |
| Glossary 상세 없음 | 짧은 정의에서 종료하거나 상위 용어 문맥·검색으로 전환 |
| 번역본 없음 | 대응 관계를 만들지 않고 대상 언어의 상위 정보공간으로 이동 |
| 오래되거나 변경된 Knowledge | 현재 유효한 Knowledge 또는 관련 최신 문맥으로 이동 |
| 예상과 다른 정보 도달 | 현재 위치를 확인하고 상위 문맥·검색·원래 문맥 중 하나로 복귀 |

## 규칙

1. 모든 예외를 전체 정보공간의 첫 진입점으로만 돌려보내지 않는다.
2. 원래 문맥이 유효하면 원래 문맥 복귀를 우선 검토한다.
3. 원래 문맥이 없거나 유효하지 않으면 가장 가까운 승인 상위 문맥을 사용한다.
4. 복구 경로는 실패를 숨기지 않으며 사용할 수 없는 관계를 명확히 구분한다.
5. 기술 오류의 원인과 처리 방법은 관련 구현 문서가 소유한다.

---

# 12. 흐름 간 충돌 및 우선순위

## 목적

한 상태에서 여러 이동 가능성이 있을 때 관계의 의미와 사용자 목적을 혼합하지 않게 한다.

## 구분 기준

| 함께 나타날 수 있는 흐름 | 구분 기준 |
|---|---|
| Previous·Next / Related | 전자는 순차 학습, 후자는 선택 확장 |
| 상위 Hub 복귀 / 다음 단계 | 전자는 주제 문맥 확인, 후자는 현재 Path 계속 |
| 검색 결과 / Discovery | 전자는 명시적 표현에 대한 탐색, 후자는 관련 Knowledge 발견 |
| Glossary / 상세 학습 | 전자는 짧은 용어 확인, 후자는 개념 학습 |
| 언어 전환 / 대응 문서 이동 | 전자는 정보공간 변경, 후자는 승인된 동일 Knowledge 관계 이동 |
| 정상 종료 / 오류 이탈 | 전자는 목적 달성 후 선택, 후자는 목적 미달성 상태의 복구 필요 |
| 필수 학습 / 선택 확장 | 전자는 선수·후속 관계, 후자는 Related 관계 |

## 우선순위 원칙

1. 현재 사용자의 명시적 목적을 주경로 판단의 첫 기준으로 삼는다.
2. 현재 질문 해결을 방해하면서 후속 탐색을 우선하지 않는다.
3. 선수 Knowledge가 현재 이해에 필수인 경우 순차 관계를 Related보다 우선한다.
4. 목적 달성 뒤에는 종료 선택을 항상 유효하게 유지한다.
5. 복구가 필요한 상태에서는 새로운 확장보다 유효한 문맥 회복을 우선한다.
6. 하나의 절대적 행동 순서를 모든 사용자에게 적용하지 않는다.

# 13. User Flow 검증 기준

## 목적

User Flow를 화면 취향이나 구현 편의가 아니라 논리적 완결성과 권한 정합성으로 검증한다.

## 검증표

| 영역 | 검증 질문 |
|---|---|
| Goal | 사용자 목적이 한 문장으로 명확한가? |
| Entry | 시작 상태와 진입 조건이 확인되는가? |
| Preconditions | 필요한 정보 객체와 관계가 승인되어 있는가? |
| Context | 이동 중 원래 목적과 현재 문맥이 유지되는가? |
| Decision | 분기 조건이 사용자 목적 또는 정보 존재 여부로 명확한가? |
| Continuity | 중간에 출구 없는 상태나 설명되지 않은 점프가 없는가? |
| Alternative | 주경로를 사용할 수 없을 때 유효한 대체 경로가 있는가? |
| Recovery | 실패 뒤 가장 가까운 유효 문맥으로 복귀할 수 있는가? |
| Success | 목적 달성 상태가 관찰 가능한 정보 상태로 정의되어 있는가? |
| Exit | 종료·복귀·확장·새 Flow 전환이 구분되는가? |
| Relationship | Previous·Next, Related, 상위 관계 및 대응 관계가 혼용되지 않는가? |
| Authority | IA와 상위 Source of Truth에 없는 관계를 만들지 않았는가? |
| Feasibility | 존재하지 않는 완료·저장·개인화 기능을 전제하지 않는가? |
| Boundary | 화면, 디자인, 구현 또는 데이터 결정을 선점하지 않았는가? |
| Handoff | Screen Architecture가 필요한 상태와 판단을 도출할 수 있는가? |

## 승인 조건

다음 조건을 모두 충족해야 승인된 Flow로 간주한다.

1. 목적, 진입, 전제조건, 성공 및 종료가 명확하다.
2. 모든 이동이 승인된 IA 관계 안에서 성립한다.
3. 정상 경로뿐 아니라 필요한 분기와 복구 경로가 존재한다.
4. 다른 Source of Truth의 정의를 복제하거나 변경하지 않는다.
5. Screen Architecture가 화면 책임을 설계할 수 있을 만큼 상태 변화가 분명하다.
6. UI·디자인·구현 방법을 포함하지 않는다.

---

# 14. Screen Architecture 전달 기준

## 목적

승인된 User Flow를 화면 체계 설계 단계로 손실 없이 전달하되 화면 결정을 선점하지 않는다.

## Flow별 전달 항목

각 승인 Flow는 다음 내용을 전달한다.

- 사용자 목적
- 진입 상태
- 전제조건
- 사용자 정보 상태의 목록
- 상태 사이의 이동
- 결정 지점과 분기 조건
- 유지해야 할 원래 문맥
- 대체 및 복구 상태
- 성공 상태
- 종료 상태
- 후속 Flow
- 참조하는 IA 객체와 관계
- 화면 단계에서 반드시 지원해야 할 사용자 판단
- 화면 단계에서 결정해야 하는 미확정 사항

## 전달하지 않는 항목

- 화면명과 화면 개수
- 별도 화면 또는 동일 화면 상태 여부
- 화면별 구성 요소
- 버튼, 카드, 메뉴 및 문구
- 화면 배치와 시각적 강조
- 기기별 표현
- 상호작용 형태
- 주소와 routing
- 코드, API 및 상태 관리 방식

## 규칙

1. User Flow는 필요한 사용자 상태를 전달한다.
2. Screen Architecture는 그 상태를 어떤 화면 체계가 담당할지 결정한다.
3. 하나의 상태가 반드시 하나의 화면을 의미하지 않는다.
4. 하나의 화면이 여러 상태를 담당할 수 있는지 여부는 이 문서에서 결정하지 않는다.
5. 화면 설계 과정에서 Flow 변경이 필요하면 이 문서의 변경 절차로 되돌아온다.

---

# 15. 변경 관리와 문서 연동

## 목적

User Flow 변경이 IA 또는 후속 설계 문서를 우회하지 않게 한다.

## 변경 순서

```text
사용자 흐름 문제 확인
→ 사용자 목적과 실패 상태 확인
→ 관련 IA 객체와 관계 확인
→ User Flow 책임 여부 판정
→ 변경안 작성
→ 사용자 승인
→ User Flow 반영
→ Screen Architecture 영향 검토
→ 디자인·구현 작업 별도 진행
```

## 변경안 필수 항목

- 현재 문제
- 영향을 받는 사용자 목적
- 영향을 받는 Flow ID
- 영향을 받는 진입·분기·복구·성공·종료 상태
- 참조하는 IA 객체와 관계
- 다른 Source of Truth 변경 필요 여부
- Screen Architecture에 미치는 영향
- 기존 직접 진입과 학습 연속성에 미치는 영향
- 변경 후 검증 결과

## 규칙

1. 승인 전에는 Flow 변경을 확정하거나 구현하지 않는다.
2. IA 관계 변경이 필요하면 IA 변경 절차를 먼저 따른다.
3. Knowledge 정의 변경이 필요하면 해당 Content Architecture의 권한을 따른다.
4. 화면 책임 변경은 Screen Architecture에서 별도로 승인한다.
5. 디자인과 구현 승인은 User Flow 변경 승인을 대신하지 않는다.
6. 기존 외부 진입, 원래 문맥 복귀 및 Learning Path의 연속성을 보호한다.
7. 변경 이력은 변경 이유와 영향을 식별할 수 있게 유지한다.

---

# 16. User Flow Anti-Patterns

## 목적

페이지 순서, 기능 목록 또는 화면 명세를 User Flow로 오인하는 것을 방지한다.

## 금지 항목

- 사이트맵을 User Flow로 사용한다.
- 페이지 목록만 화살표로 연결한다.
- 모든 사용자가 홈페이지에서 시작한다고 가정한다.
- 모든 사용자를 하나의 연속 경로로 강제한다.
- Category, Hub, Topic, Node의 관계를 이 문서에서 다시 정의한다.
- IA에 없는 이동 관계를 임의로 추가한다.
- Previous·Next와 Related를 같은 이동으로 처리한다.
- 상위 Hub 복귀와 다음 학습 단계를 혼용한다.
- 검색 결과와 Discovery를 같은 흐름으로 처리한다.
- Glossary 확인을 상세 학습과 동일하게 취급한다.
- 언어 정보공간 이동과 대응 번역 이동을 혼용한다.
- 실패, 대체 또는 복귀 경로를 생략한다.
- 모든 실패를 전체 정보공간의 첫 진입점 복귀로만 해결한다.
- 목적 달성 후 종료 상태 없이 계속 탐색시킨다.
- 실제로 없는 완료, 저장, 개인화 또는 학습 재개 상태를 가정한다.
- 버튼, 카드, 메뉴, 탭 또는 화면 위치로 흐름을 설명한다.
- 화면명, 화면 수, 화면 구성 요소 또는 레이아웃을 확정한다.
- URL, routing, 코드 또는 데이터 구조를 흐름 규칙으로 사용한다.
- 검색 또는 추천 알고리즘을 정의한다.
- 디자인 헌법의 시각 원칙을 반복한다.
- 사용자 감정을 새로운 시스템 상태로 과도하게 세분화한다.
- 구현 편의를 이유로 승인된 사용자 목적이나 IA 관계를 왜곡한다.

---

# 17. 최종 불변 원칙

## 목적

Robo5 User Flow가 장기적으로 보호해야 할 고유 책임을 최종 선언한다.

## 원칙

```text
Information Architecture는
사용자가 이동할 수 있는 Knowledge 공간을 정의한다.

User Flow는
그 공간 안에서 사용자가 목적을 달성하기까지의
진입, 상태 변화, 선택, 분기, 복귀와 종료를 정의한다.

Screen Architecture는
각 상태와 이동을 담당하는 화면 체계를 정의한다.

Design은
승인된 흐름과 화면 책임을 사용자가 이해할 수 있게 표현한다.

Implementation은
승인된 구조와 흐름을 손상 없이 실행한다.
```

## 규칙

1. User Flow는 페이지 순서가 아니라 목적별 상태 변화다.
2. 모든 Flow는 승인된 IA 안에서만 성립한다.
3. 현재 질문 해결과 종료 가능성을 보호한다.
4. 순차 학습과 선택 확장을 구분한다.
5. 문맥 이탈 뒤 복귀 경로를 보호한다.
6. 실패 상태에서 존재하지 않는 관계를 만들지 않는다.
7. 실제로 존재하지 않는 사용자 상태를 전제하지 않는다.
8. 화면, 디자인 및 구현의 결정을 선점하지 않는다.
9. User Flow 변경은 기존 Source of Truth의 권한과 승인 절차를 따른다.

Robo5 User Flow는 사용자를 오래 머물게 만드는 경로가 아니다.

사용자가 자신의 목적을 이해하고, 유효한 Knowledge에 도달하며, 필요한 만큼 학습하고, 길을 잃지 않은 채 종료하거나 다음 목적을 선택하게 하는 이동 기준이다.

---

# 18. Screen Architecture Handoff

## 목적

이 문서의 책임을 명확히 종료하고 다음 공식 설계 문서로 인계한다.

## User Flow의 최종 산출 범위

User Flow는 다음까지 정의한다.

```text
사용자 목적
→ 진입 상태
→ 전제조건
→ 사용자 정보 상태
→ 상태 변화
→ 선택과 분기
→ 대체와 복구
→ 성공 상태
→ 종료 상태
→ 후속 Flow
```

## 후속 문서의 책임

각 상태를 어떤 화면 체계가 담당할 것인지는 `ROBO5_SCREEN_ARCHITECTURE.md`에서 정의한다.

Screen Architecture는 이 문서에 승인된 상태와 이동을 입력으로 사용하며 다음 질문에 답해야 한다.

- 각 사용자 상태를 어떤 화면 책임이 지원하는가?
- 상태 사이의 이동을 어떤 화면 체계가 이어 주는가?
- 여러 진입 상태를 동일한 화면 책임이 수용할 수 있는가?
- 대체·복구·빈 상태를 어떤 화면 책임이 담당하는가?
- 사용자가 현재 문맥과 다음 선택을 판단하기 위해 각 화면이 어떤 정보 책임을 가져야 하는가?

## 인계 규칙

1. Screen Architecture는 User Flow에 없는 사용자 목적이나 상태를 임의로 추가하지 않는다.
2. 새로운 상태가 필요하면 먼저 User Flow 변경 필요성을 검토한다.
3. 새로운 정보 객체나 관계가 필요하면 먼저 IA의 권한을 확인한다.
4. Screen Architecture는 화면 책임을 정의하되 디자인과 구현 방법을 확정하지 않는다.
5. User Flow의 성공·종료·복구 조건은 화면 설계 과정에서도 유지한다.
6. 직접 진입, 순차 학습, 선택 확장, 용어 확인, 언어 이동 및 실패 복구의 차이를 화면 책임으로 전달한다.
7. `ROBO5_SCREEN_ARCHITECTURE.md`의 승인은 별도 절차로 진행한다.

이 문서의 다음 공식 설계 단계는 `ROBO5_SCREEN_ARCHITECTURE.md`다.
